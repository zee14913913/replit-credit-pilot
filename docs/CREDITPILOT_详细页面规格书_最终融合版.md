### 设计理念
**为什么将RECEIPTS和INVOICES整合到CREDIT CARDS？**

1. **业务逻辑自然关联**: 
   - 收据 → 验证信用卡交易
   - 发票 → 源自信用卡供应商交易
   - 三者本质上是同一财务流程的不同环节

2. **用户工作流优化**:
   - 传统：上传账单 → 切换到Receipts → 切换到Invoices → 切换回Credit Cards → 切换到Reports
   - 现在：在Credit Cards一个页面完成所有操作

3. **数据完整性提升**:
   - 实时对账：账单数据 ↔ 收据数据 ↔ 发票数据
   - 即时反馈：覆盖率、匹配状态、评分一目了然

#### OCR提取字段
```python
{
    "merchant_name": "DINAS RESTAURANT",
    "amount": 85.00,
    "date": "2025-11-05",
    "time": "14:30",
    "receipt_number": "INV-20251105-001",
    "confidence_score": 0.85  # 信心评分
}
```

#### 匹配算法
```
匹配相似度 = 综合匹配度（各维度 × 权重系数）

权重分配：
- 金额相似度（权重40%）
- 日期相似度（权重30%）
- 商户名相似度（权重20%）
- 其他特征（权重10%）

评级标准：
- 优秀: 整体匹配良好，可自动确认
- 良好: 高度匹配，建议人工确认
- 一般: 部分匹配，需仔细核对
- 差: 匹配度低，建议重新匹配
```

## 数据分类系统

### OWNER vs INFINITE 6字段分类

```python
# 月度账单6大字段
monthly_statements = {
    # OWNER (客户个人)
    "owner_expenses": Decimal("8500.00"),   # 客户消费
    "owner_payments": Decimal("6000.00"),   # 客户还款
    "owner_credits": Decimal("0.00"),       # 客户收入
    
    # INFINITE (7家指定供应商)
    "gz_expenses": Decimal("3200.00"),      # INFINITE消费
    "gz_payments": Decimal("2800.00"),      # INFINITE还款
    "gz_credits": Decimal("0.00"),          # INFINITE收入
}

# 服务费收入计算
service_fee = gz_expenses * 0.01  # 1%
```

### 分类逻辑

```python
def classify_transaction(description, amount):
    """
    分类交易为OWNER或INFINITE
    """
    # 7家指定供应商关键词
    INFINITE_SUPPLIERS = [
        "7SL", "DINAS", "RAUB", "HAINAN", "AI SMART",
        "HUAWEI", "华为", "PASAR RAYA", "PUCHONG HERBS"
    ]
    
    for supplier in INFINITE_SUPPLIERS:
        if supplier.lower() in description.lower():
            if amount > 0:  # 消费
                return "gz_expenses"
            else:  # 还款
                return "gz_payments"
    
    # 默认为OWNER
    if amount > 0:
        return "owner_expenses"
    else:
        return "owner_payments"
```
